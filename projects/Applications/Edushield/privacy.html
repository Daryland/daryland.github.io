Overview

Privacy-first by design. EduShield is a desktop application. All submission data, student records, and analysis results are stored locally in a SQLite database on your machine. No submission content ever leaves your institution unless you explicitly share a report.

This Privacy Policy explains what information EduShield collects, how it is used, and how it is protected. By installing and using EduShield, you agree to this policy.

Data We Collect

EduShield is an offline-first application. The categories of data it handles are:

Account Data (stored locally)

  • Institutional email address used to log in
  • Hashed password (bcrypt — never stored in plaintext)
  • User role (Teacher, Admin, Super Admin)
  • License tier (FREE, DEPARTMENT, CAMPUS)

Submission Data (stored locally)

  • Student name and institutional ID
  • Submitted text or file content
  • Grade level and assignment context
  • AI detection scores and signal breakdowns
  • Timestamp and submitting teacher identity

Usage Telemetry

EduShield does not currently collect anonymous usage telemetry. No data is sent to EduShield servers during normal operation.

Local Storage

All data is stored in a SQLite database at the following location, depending on your operating system:

PlatformDatabase Path
Windows%APPDATA%\EduShield\edushield.db
macOS~/Library/Application Support/EduShield/edushield.db
Linux~/.local/share/EduShield/edushield.db

Your institution is responsible for applying appropriate file-system permissions to protect this database. EduShield does not encrypt the database file by default — we recommend using OS-level disk encryption (BitLocker, FileVault) on machines running EduShield.

AI Processing

EduShield uses two distinct AI mechanisms:

Local detection engine

The primary detection analysis (perplexity scoring, burstiness, vocabulary richness, hedging patterns, structural templating, personal voice) runs entirely on-device using a statistical language model. No text is transmitted externally for this analysis.

Socratic Verification (Anthropic API)

Network request required. When a teacher triggers the Socratic Verification feature, the submission text is sent to Anthropic's API to generate follow-up questions. This is the only scenario in which submission content leaves your machine.
  • Data sent: submission text content only
  • Provider: Anthropic, PBC (claude-3 model family)
  • Anthropic's data handling is governed by their Privacy Policy
  • You can disable this feature by not configuring an ANTHROPIC_API_KEY

Submission Data

Submission text and analysis results belong to your institution. EduShield:

  • Does not transmit submission content to its own servers
  • Does not use submissions to train AI models
  • Does not share student work with third parties (except Anthropic, when Socratic Verification is used)
  • Allows bulk export and deletion of submission records at any time

PDF reports generated by EduShield contain submission analysis data. These reports are created locally and saved to your machine. Sharing reports is your institution's responsibility.

Student Privacy

EduShield is designed to comply with student privacy regulations including FERPA (US) and similar frameworks. Key protections:

  • Student records are accessible only to authenticated, role-authorized users within your institution
  • No student data is shared with external services (except as noted in the AI Processing section)
  • Student data does not leave your institution's hardware without deliberate action (e.g., sharing a PDF report)
  • Institutions are the data controller for all student records stored in EduShield
EduShield stores student names and IDs as entered by teachers. Do not enter more personally identifiable information than is operationally necessary.

Data Retention

Submission records and user accounts persist until explicitly deleted. EduShield does not automatically purge data. Administrators can:

  • Delete individual submissions from the dashboard
  • Deactivate or remove user accounts from the Admin panel
  • Remove all data by deleting the SQLite database file

Institutions should define their own retention schedules consistent with their policies and applicable regulations.

Your Rights

Because all data is stored locally on institution-controlled hardware, your institution has full control over all records. Depending on your jurisdiction, individuals may have rights to:

  • Access records stored about them
  • Request correction of inaccurate data
  • Request deletion of their data

Direct requests to your institution's EduShield administrator. EduShield (the software) has no independent access to your data.

Security

  • Passwords are hashed with bcrypt (cost factor 12) before storage
  • Session tokens are signed JWTs with expiration
  • API routes enforce role-based authorization on every request
  • Rate limiting is applied to authentication endpoints
  • The local API server binds to localhost only — not exposed to the network

No system is perfectly secure. We recommend keeping EduShield updated, restricting access to the database file, and enabling OS-level disk encryption.

Policy Changes
